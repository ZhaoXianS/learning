    设计模式之美

    Khaki 2022/8/18 设计模式

    # 学习目标：

    • [ ] 设计模式

    # 学习内容：

    # 1.代码设计的重要性

    当我们遇到一个与业务无关的功能模块的开发任务时，面对这样的一个稍复杂的代码的设计 和开发任务时我们经常会感到无从下手，这时就体现出了代码设计的重要性。

    # 2.代码设计原则：先有问题，后有方案。不要脱离具体的场景谈代码设计。

    # 3.面向对象分析、面向对象设计、面向对象编程

    面向对象设计步骤：

    （1）划分职责进而识别有哪些类：①进行模块划分②拆解成功能点列表

    （2）定义类及其属性和方法

    （3）定义类之间的交互关系：①泛化②实现③组合

    （4）将类组装起来并提供执行入口

    # 4.基于“贫血”模型的传统开发模式和基于“充血”模型的DDD（领域驱动设计）开发模式

    “贫血”模型将数据与操作分离，破换了面向对象编程的封装特性，属于典型的面向过程编程风格。

    例如：UserBo只含数据，UserService只含业务，两者都在service层。

    “充血”模型的DDD开发，service层包含Service和Domain，Domain类既包含数据又包含业务。

    基于“贫血”模型的传统开发模式，重Service类，轻BO类；基于“充血”模型的DDD开发模式轻Service，中Domain类。

    # 5.抽象类侧重代码复用，接口侧重解耦，接口定义一种协议或者契约。

    # 6.基于接口而非实现类编程

    # 7.组合优于继承

    # 8.设计原则（具体细节参考书籍）

    单一职责原则：一个类或模块只负责完成一个职责。

    开闭原则(看源码)：对扩展开放，对修改关闭。

    里氏替换原则：子类对象能够替换到程序中父类对象出现的任何地方，并且保证程序原有的逻辑行为布标和正确性不被破坏。

    接口隔离原则：客户端不应该被强迫依赖它不需要的接口。

    依赖反转原则：实体必须依靠抽象而不是具体实现。它表示高层次的模块不应该依赖于低层次的模块，它们都应该依赖于抽象。

    KISS（尽量保持简单）原则和YAGNI（不要设计当前用不到的功能，）原则，KISS原则讲的是如何做，YAGNI原则讲的是要不要做。

    DRY（不要编写重复的代码）原则，LOD（最少知道原则）原则。

    # 9.代码规范

    （1）模块化思维，善于将大块的复杂业务的代码封装成类或函数。

    （2）当参数超过5个时，尽量封装成类。

    （3）减少嵌套层数，将部分嵌套代码封装成函数。

    # 10.重构技巧

    1.小型重构主要是针对类、函数和变量等级别的重构，如规范命名、规范注释、消除超大类或函数、提取重复代码，所以说小的重构随时都可以进行。

    2.大型重构中保证重构不出错的有效手段就是编写单元测试

    3.编写可测试的代码

    4.解耦的方法：

    （1）通过封装与抽象进行解耦

    （2）引入中间层

    （3）模块化

    （4）利用代码设计思想和设计原则

    5.当函数抛出异常时，我们应该选择哪种方式

    异常是否抛给上层调用函数，取决于上层函数是否与发生的异常有相关性，是否封装成新的异常抛给上层函数取决于上层调用函数能不能处理这个异常。

    # 11.创建型设计模式（具体细节参考书籍）

    单例模式

    如果某个类并没有拓展的需求，并且不依赖外部系统，那么，为了限制其对象的全局唯一性，将其设计出成单例就是合理的，spring注入的对象就是单例实现的。

    ‘饿汉’式：线程安全，不支持延迟加载。

    ‘懒汉’式：支持延迟加载。

    双重检测

    建造者模式:‘定制化’创建不同的对象

    原型模式：利用已有对象（原型）进行复制（拷贝）的方式进行创建对象。

    # 12.结构型设计模式（具体细节参考书籍）

    代理模式（看源码）：在不改变原始类的情况下，通过引入代理类来给原始类附加不相关的其他功能。

    应用：SpringAOP、日志、幂等、事物、鉴权、监控、统计、限流、RPC（远程代理）、缓存中的应用。

    装饰器模式：用组合代替继承，给原始类添加了相关的其他功能。

    适配器模式

    应用：SLF4J对不同的日志框架的接口进行二次封装，适配成统一的SLF4J接口定义。

    桥接模式：组合代替继承，将MxN的继承关系变成M+N的组合关系

    门面模式：为子系统定义一组统一的接口

    # 13.行为型设计模式（具体细节参考书籍）

    观察者模式（发布/订阅模式）

    同步阻塞的观察者模式主要是为了代码解耦；异步非阻塞的观察者模式除了能够实现代码解耦外，还能够提高代码的执行效率；跨进程的观察者模式一般基于消息队列，其解耦更加彻底，用来实现不通进程间的被观察者和观察者的交互。

    模板方法模式：在一个方法中定义一个算法框架，并将某些步骤推迟到子类中实现

    作用：复用和扩展

    回调：同步回调指在函数返回之前执行回调函数，异步回调是指在函数返回之后执行回调函数，同步回调看起来像模板方法模式，异步回调看起来像观察者模式。回调比模板方法模式更加灵活。

    策略模式：定义一组算法，将每个算法分别封装，让他们可以互相替换。

    策略模式的主要作用是解耦策略的定义、创建和使用，控制代码的复杂度，让每个部分都不太复杂。

    应用：支付方式、电商网站活动。

    职责链模式：将请求的发送和接收解耦，让多个接收对象都有机会处理这个请求；将这些接收对象串成一条链，并沿着这条链传递这个请求，直到链上的某个接收对象能够处理它为止。

    应用：拦截器、过滤器。
